2   STRIDE ELEMENT DEEP DIVE Threat Scenarios, Attack Vectors, System Locations, and Mitigations for Each Element
S

SPOOFING

Property Violated: Authentication
"Can an attacker pretend to be someone or something else?"

⚠ Attack Vectors

  • Credential stuffing & brute-force attacks
  • Phishing / social engineering campaigns
  • Session hijacking & token replay
  • Man-in-the-Middle (MITM) interception
  • Certificate forgery / DNS spoofing
  • API key theft & OAuth token manipulation

📍 Where in System

TB-0→TB-1 DF-1/DF-2 Auth Service Admin Console CI/CD Cache Store

✓ Mitigations

  • Multi-Factor Authentication (MFA) at all entry points
  • Mutual TLS (mTLS) for service-to-service communication
  • Strong password policies + account lockout mechanisms
  • OAuth 2.0 / OIDC federation with short-lived JWT tokens
  • Certificate pinning + DNSSEC
  • API key rotation + IP allowlisting for admin access
  • Hardware Security Modules (HSM/TPM)
T

TAMPERING

Property Violated: Integrity
"Can an attacker modify data in transit, at rest, or in processing?"

⚠ Attack Vectors

  • SQL / NoSQL injection attacks
  • Cross-Site Scripting (XSS) – stored/reflected
  • Parameter & request tampering
  • Man-in-the-Middle data modification
  • XML/JSON injection & deserialization attacks
  • Supply chain attacks (CI/CD pipeline poisoning)
  • Log injection / truncation
  • Binary patching & code injection

📍 Where in System

ALL Data Flows TB-2→TB-3 Database File Storage Message Queue CI/CD SIEM/Logs

✓ Mitigations

  • Input validation + output encoding (parameterized queries)
  • Digital signatures + HMAC on all critical data flows
  • TLS 1.3 everywhere (encrypt in transit)
  • Database integrity constraints + checksums
  • Code signing for CI/CD artifacts + SBOM
  • Immutable infrastructure (containers + IaC)
  • Write-once append-only log storage (WORM)
  • File Integrity Monitoring (FIM)
R

REPUDIATION

Property Violated: Non-Repudiation
"Can an attacker deny performing an action without being disproved?"

⚠ Attack Vectors

  • Log deletion / manipulation / injection
  • Insufficient or absent audit logging
  • Using shared / generic accounts
  • Timestamp manipulation & clock skew exploitation
  • Anti-forensics techniques
  • Claiming session was hijacked (plausible deniability)

📍 Where in System

DF-1/DF-2 (User actions) Auth Service Business Logic Admin Console CI/CD SIEM/Logs Message Queue Database

✓ Mitigations

  • Comprehensive audit logging (Who, What, When, Where, From Where)
  • Centralized SIEM with tamper-evident, write-once storage
  • Digital signatures on critical transactions
  • NTP synchronization for accurate timestamps
  • Unique identity per user/service (no shared accounts)
  • Database transaction logging (CDC / triggers)
  • Secure Timestamping Authority (TSA)
  • Blockchain / Merkle tree based audit chains
I

INFORMATION DISCLOSURE

Property Violated: Confidentiality
"Can an attacker access data they are not authorized to see?"

⚠ Attack Vectors

  • Network sniffing / eavesdropping
  • Directory traversal & path manipulation
  • IDOR (Insecure Direct Object Reference)
  • Verbose error messages / stack trace exposure
  • Side-channel & timing attacks
  • Memory scraping (Heartbleed-type vulnerabilities)
  • Cloud storage misconfiguration (public buckets)
  • API over-exposure (excessive data in responses)
  • Backup exposure & credential leakage

📍 Where in System

DF-1/DF-2 (if weak encryption) DF-8/DF-9/DF-10 Database File Storage Cache Store SIEM/Logs Load Balancer (headers) Error Responses

✓ Mitigations

  • Encryption at rest (AES-256) + in transit (TLS 1.3)
  • Data classification + labeling + masking/tokenization
  • Data Loss Prevention (DLP) at network + host level
  • Secrets management (Vault / KMS / HSM)
  • Least privilege data access on need-to-know basis
  • API response filtering + field-level encryption
  • Secure error handling (no stack traces or debug info)
  • Network segmentation + micro-segmentation
D

DENIAL OF SERVICE

Property Violated: Availability
"Can an attacker crash the system or make it unusable?"

⚠ Attack Vectors

  • Volumetric DDoS (UDP/ICMP/SYN floods)
  • Application-layer DoS (Slowloris, HTTP floods)
  • Resource exhaustion (CPU / Memory / Disk / Connections)
  • Algorithmic complexity attacks (Hash DoS)
  • XML bomb / Billion Laughs attack
  • ReDoS (Regular Expression Denial of Service)
  • Zip bombs / decompression bombs
  • Cascading failure / thundering herd

📍 Where in System

TB-0→TB-1 (Internet→Perimeter) Firewall/WAF Load Balancer Web App / API GW Auth Service (lockout) Database (connections) Message Queue File Storage (disk full)

✓ Mitigations

  • CDN + DDoS protection (Cloudflare / Akamai / AWS Shield)
  • Rate limiting + throttling at API GW + WAF + per-user quotas
  • Auto-scaling + load balancing + multi-AZ redundancy
  • Circuit breaker pattern + bulkhead isolation
  • Resource quotas + limits (CPU/Mem/Disk/Connections)
  • Connection pooling + timeout configuration
  • Health checks + monitoring + alerting + auto-recovery
  • Graceful degradation + failover strategies
E

ELEVATION OF PRIVILEGE

Property Violated: Authorization
"Can an attacker gain capabilities they should not have?"

⚠ Attack Vectors

  • Broken access control (OWASP #1)
  • JWT / token manipulation & IDOR exploitation
  • Role parameter tampering
  • Kernel / OS privilege escalation exploits
  • Container escape (Docker breakout)
  • Dependency / library vulnerabilities
  • Misconfigured RBAC / ABAC policies
  • SQL injection → OS command execution
  • Insecure deserialization → Remote Code Execution

📍 Where in System

Auth/AuthZ Service Web App / API GW Business Logic Tier Admin Console CI/CD Pipeline TB-2→TB-4 Database (stored procs)

✓ Mitigations

  • Principle of Least Privilege (PoLP) for all accounts + services
  • Role-Based + Attribute-Based Access Control (RBAC + ABAC)
  • Server-side authorization enforcement (never trust client)
  • Regular privilege audit + access reviews
  • Container hardening + non-root execution
  • Sandboxing + Seccomp + AppArmor / SELinux profiles
  • Vulnerability management + patch management + dependency scanning
  • Zero Trust Architecture (verify every request)
3   THREAT MODELING LIFECYCLE PROCESS Six-Phase Iterative Methodology for Continuous Threat Assessment
1

System Decomposition

  • Identify all assets & components
  • Map data flows (DF-1…DF-n)
  • Define trust boundaries
  • Create DFD (Level 0, 1, 2)
  • Identify entry/exit points
  • Catalog data sensitivity
→
2

Threat Enumeration

  • Apply STRIDE per element
  • External Entities → S, R
  • Processes → S,T,R,I,D,E
  • Data Stores → T, I, D
  • Data Flows → T, I, D
  • Trust Boundary crossings
→
3

Risk Assessment

  • Likelihood × Impact scoring
  • DREAD or CVSS rating
  • Prioritize by risk level
  • Critical / High / Med / Low
  • Map to business impact
  • Regulatory implications
→
4

Mitigation Strategy

  • MITIGATE → Apply controls
  • TRANSFER → Insurance/outsource
  • AVOID → Remove feature
  • ACCEPT → Document risk
  • Defense-in-depth layering
  • Control effectiveness testing
→
5

Validation

  • Penetration testing per STRIDE
  • Security code review
  • SAST / DAST / SCA scanning
  • Red team exercises
  • Threat model review board
  • Compliance validation
→
6

Continuous Monitoring

  • SIEM alerts → STRIDE mapping
  • Update model per release
  • Threat intel integration
  • Incident response feedback
  • Compliance audit alignment
  • ↩ Loop back to Phase 1
4   STRIDE SECURITY PROPERTIES CROSS-REFERENCE Mapping Each Element to Security Principle, Desired State, and Verification Method
STRIDE Element Security Property Violated Principle Desired State Verification Method
S Spoofing Authentication Identity verification bypassed Every entity verified before access granted Auth penetration testing, credential stuffing tests, MFA audit
T Tampering Integrity Data or code modified without detection All data verified for integrity at every boundary FIM, checksum validation, SAST/DAST, injection testing
R Repudiation Non-Repudiation Actions cannot be attributed to an actor Every action is attributable & provable Log review, audit trail verification, forensic readiness assessment
I Info Disclosure Confidentiality Unauthorized data exposure Data accessible only to authorized parties DLP testing, encryption validation, access control reviews
D Denial of Service Availability System/service becomes unavailable System remains available under adverse conditions Load testing, chaos engineering, DDoS simulation, failover tests
E Elevation of Privilege Authorization Unauthorized capability gain Users/services operate only within granted scope Privilege escalation testing, RBAC review, container breakout tests

STRIDE Threat Modeling Framework — Originally developed by Microsoft
This diagram provides a repeatable, auditable framework for systematically identifying, assessing, and mitigating threats across every layer of a general enterprise system.
Threat modeling is iterative: every architecture change, deployment, or incident should trigger a threat model review cycle.
